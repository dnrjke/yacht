# 코드베이스 위생 점검 (Codebase Audit)

지정 범위(기본: 전체 워크스페이스)를 순차 점검하여 죽은 코드, 타입 이슈, 공통화/리팩터링 대상을 찾아 보고한다.

## 사용법

```
/yacht-audit                    ← 전체 (frontend + backend + core)
/yacht-audit frontend           ← frontend만
/yacht-audit backend            ← backend만
/yacht-audit core               ← core만
/yacht-audit frontend/src/store ← 특정 경로
```

## 실행 절차

### 1단계: 범위 결정 + tsc

1. 인자가 있으면 해당 경로/워크스페이스, 없으면 frontend·backend·core 전체를 대상으로 한다.
2. 각 워크스페이스에서 `tsc --noEmit` 실행 → 타입 에러가 있으면 수집해 둔다 (점검 결과에 포함).
   - `cd C:/yacht/core && npx tsc --noEmit`
   - `cd C:/yacht/frontend && npx tsc --noEmit`
   - `cd C:/yacht/backend && npx tsc --noEmit`

### 2단계: 모듈 분할

대상 범위의 파일을 **모듈 단위**로 분할한다. 분할 기준:

```
frontend/src/components/screens/ → screens (GameScreen, MainMenuScreen 등)
frontend/src/components/ui/      → ui (Scoreboard 등)
frontend/src/components/3d/      → 3d (PhysicsBoard, PhysicsCup, PhysicsDice 등)
frontend/src/store/              → store (gameStore)
frontend/src/                    → frontend-root (App.tsx, main.tsx)
backend/src/                     → backend (server.ts)
backend/src/physics/             → physics (PhysicsWorld)
core/src/                        → core (scoring, index)
```

인자로 단일 모듈을 지정한 경우 분할 없이 해당 모듈만 점검.

### 3단계: 순차 점검

모듈을 **하나씩 순차적으로** 점검한다 (토큰 절약을 위해 병렬 서브에이전트 사용하지 않음).

각 모듈에 대해 **직접** 다음을 확인:

#### A. 죽은 코드
- 프로젝트 내 어디서도 import/참조되지 않는 export (함수, 클래스, 상수, 타입)
- 도달 불가능한 분기 (항상 true/false인 조건)
- 주석 처리된 코드 블록
- 빈 함수/메서드 (stub으로 남겨진 것)

#### B. 타입 이슈
- `any` 사용 (명시적/암묵적)
- 불필요한 타입 단언 (`as`, `!`)
- 제네릭 누락 (예: `Array` → `Array<T>`)
- `// @ts-ignore` / `// @ts-expect-error`

#### C. 공통화/추상화 대상
- 2개 이상 모듈에서 반복되는 유사 코드 패턴 (3줄 이상)
- 동일 목적의 유틸 함수가 여러 파일에 각각 정의된 경우
- 상수/매직넘버가 여러 곳에 하드코딩된 경우
- frontend/backend 간 공유되어야 할 타입이 core를 경유하지 않는 경우

#### D. 리팩터링 후보
- 과대 파일 (500줄 이상이면서 책임이 2개 이상)
- 깊은 중첩 (3단 이상 if/for)
- 긴 매개변수 목록 (5개 이상)
- God object (너무 많은 필드/메서드를 가진 클래스)

#### E. 아키텍처 원칙 위반

점검 시작 전 `Docs/implementation_plan.md`를 읽고, 아키텍처 결정 전체를 체크리스트로 사용한다:

**모노레포 워크스페이스 경계**
- 게임 로직(점수 계산 등)이 core가 아닌 frontend/backend에 직접 구현되어 있는지
- frontend/backend 간 공유 타입이 core를 경유하지 않고 각자 정의하는지
- UI 로직이 backend에, 서버 로직이 frontend에 위치하는지

**서버 권위(Server-Authoritative) 원칙**
- 클라이언트에서 게임 결과(주사위 값, 점수)를 직접 결정하는 코드
- 서버 검증 없이 클라이언트 상태만으로 게임 진행을 결정하는 흐름
- 물리 시뮬레이션 결과를 클라이언트가 서버에 보고하는 역전 패턴

**결정론적 물리 시뮬레이션**
- 서버와 클라이언트의 물리 시뮬레이션이 동기화 전략을 따르지 않는 부분
- Rapier 3D WASM 물리 연산에서 비결정적 요소 (랜덤, 부동소수점 불일치)
- 물리 상태 스냅샷/보간이 implementation_plan.md 전략과 불일치

**Zustand 스토어 패턴**
- gameStore.ts의 기존 패턴과 불일치하는 상태 관리 코드
- 스토어 외부에서 직접 상태를 변경하는 코드
- 불필요한 구독 (셀렉터 없이 전체 스토어 구독)

**React/R3F 패턴**
- Three.js 명령형 코드가 R3F 선언적 패턴 대신 사용된 부분
- 컴포넌트 디렉터리 구분(screens/ui/3d) 위반
- useFrame 내 과도한 연산 (프레임 드롭 원인)

**Socket.io 이벤트 구조**
- 기존 이벤트 네이밍 패턴과 불일치하는 이벤트
- 에러 핸들링 누락된 소켓 이벤트
- 연결 해제 시 정리(cleanup) 누락

> **운용**: `Docs/` 디렉터리에 아키텍처 문서가 추가되면 점검 시마다 최신 문서를 읽어 위 목록에 없는 신규 원칙도 반영할 것.

### 4단계: 심각도 분류

모든 발견 사항을 **심각도**와 **확신도** 두 축으로 분류한다.

#### 심각도 (수정 순서 결정)

| 등급 | 기준 | 예시 | 수정 여부 |
|------|------|------|----------|
| **P1 크리티컬** | 빌드 실패, 런타임 에러 가능성 | tsc 에러, 서버 권위 위반으로 치팅 가능, 잘못된 타입 단언으로 런타임 크래시 | 즉시 수정 |
| **P2 수정 권장** | 코드 품질 저하, 명백한 문제 | 명시적 `any`, `ts-ignore`, 불필요한 타입 단언, 워크스페이스 경계 위반 | 수정 |
| **P3 통합 권장** | 중복/분산으로 유지보수 비용 증가 | 매직넘버 하드코딩, 동일 유틸 중복 정의, core 미경유 타입 공유 | 수정 |
| **P4 리팩터링 권장** | 구조적 개선 대상 | 과대 파일 분리, 깊은 중첩, R3F 선언적 패턴 미준수 | 수정 |
| **P5 보류** | 확신 부족 또는 부작용 판단 필요 | 미사용 의심 export, 추정 죽은 코드, 대규모 구조 변경 | 보고만 |

#### 확신도

- **확실** — Grep/Read로 완전히 확인됨. 부작용 없이 수정 가능.
- **추정** — 높은 개연성이나 동적 참조·간접 의존 가능성이 남음. 추정 사유 필수 기재.

**P5 보류 분류 기준** — 다음 중 하나라도 해당하면 P5:
- 확신도가 "추정"인 죽은 코드 (동적 참조 가능성, 짧은 이름, re-export 체인)
- 의도적으로 남겨둔 코드일 가능성 (미래 사용 예정, 플래그 뒤 비활성 등)
- 수정 시 부작용 범위를 확정할 수 없는 경우
- 대규모 구조 변경이 필요한 리팩터링

### 5단계: 결과 파일 작성

`Docs/Audit/YYYY-MM-DD.md`에 심각도별로 분리하여 작성한다.

```markdown
# 코드베이스 점검 결과 — YYYY-MM-DD

## 점검 범위
(대상 워크스페이스, 파일 수, 총 줄 수)

## tsc 결과
(에러 있으면 워크스페이스별 목록, 없으면 "에러 없음")

## P1 크리티컬
| # | 파일:줄 | 종류 | 설명 | 확신도 |
|---|---------|------|------|--------|

## P2 수정 권장
| # | 파일:줄 | 종류 | 설명 | 확신도 |
|---|---------|------|------|--------|

## P3 통합 권장
| # | 관련 파일 | 패턴 | 제안 | 확신도 |
|---|----------|------|------|--------|

## P4 리팩터링 권장
| # | 파일 | 사유 | 제안 | 확신도 |
|---|------|------|------|--------|

## P5 보류
| # | 파일 | 대상 | 설명 | 보류 사유 |
|---|------|------|------|----------|

## 요약
- P1 크리티컬: N건
- P2 수정 권장: N건
- P3 통합 권장: N건
- P4 리팩터링 권장: N건
- P5 보류: N건 (수정 안 함)
```

같은 날짜에 이미 파일이 있으면 `-2`, `-3` 접미사를 붙인다.

### 6단계: 터미널 보고 + 수정 대기

결과 파일 작성 후, **터미널에** 요약을 보고한다:

```
## 점검 완료

- P1 크리티컬: N건
- P2 수정 권장: N건
- P3 통합 권장: N건
- P4 리팩터링 권장: N건
- P5 보류: N건 (수정 안 함)

결과 파일: Docs/Audit/YYYY-MM-DD.md

"수정 진행" → P1부터 순차 수정 시작
```

코드를 수정하지 않고 사용자 지시를 기다린다.

### 7단계: 순차 수정 (사용자 "수정 진행" 지시 시)

P1 → P2 → P3 → P4 순서로 한 등급씩 수정한다. **P5는 수정하지 않는다.**

각 등급에서:

1. **직전 재검증 (JIT)**: 해당 등급의 항목 각각을 수정 직전에 다시 Grep/Read로 확인한다.
   - 점검 시점과 수정 시점 사이에 코드가 변경되었을 수 있음.
   - 재검증 실패 시 해당 항목을 스킵하고 사유를 보고.
2. **수정 실행**: 재검증 통과한 항목만 수정.
3. **tsc 게이트**: 각 워크스페이스에서 `tsc --noEmit` 실행.
   - 통과 → 다음 등급으로.
   - 실패 → 해당 등급 수정에서 발생한 문제를 진단·수정한 뒤 다시 tsc. 해결 안 되면 해당 등급 수정을 롤백하고 사용자에게 보고.
4. **등급 완료 보고**: 수정 내역 + tsc 결과를 터미널에 보고.

```
P1 수정 완료 (N건 수정, M건 스킵) → tsc ✓
  ↓
P2 수정 완료 (N건 수정, M건 스킵) → tsc ✓
  ↓
P3 수정 완료 (N건 수정, M건 스킵) → tsc ✓
  ↓
P4 수정 완료 (N건 수정, M건 스킵) → tsc ✓
  ↓
전체 완료 보고 + P5 보류 목록 리마인드
```

## 신뢰성 규칙

### 확인된 사실만 보고
- Grep/Read로 **실제 확인한 것만** 보고한다. 추측 금지.
- "아마 안 쓰일 것 같다"는 보고하지 않는다. Grep 결과가 0건이어야 "참조 없음".

### 서브에이전트 결과 교차 검증
- 컨텍스트 한계로 서브에이전트를 사용한 경우, 보고된 **모든 발견 사항을 본 에이전트가 Grep/Read로 재확인**한 뒤에만 결과에 포함한다.
- 서브에이전트가 보고했으나 재확인에서 불일치하면 결과에서 제외한다.

### 직전 재검증 (Just-in-Time Verification)
- 수정은 발견 시점이 아닌 **수정 직전 시점**의 코드 상태를 기준으로 한다.
- 점검과 수정 사이에 다른 작업이 끼어들 수 있으므로, 수정 지시를 받으면 반드시 해당 항목을 다시 확인한 뒤 진행한다.

### 미사용 코드는 보수적으로
- 미사용 의심 export는 확실한 근거 없이 삭제하지 않는다.
- 동적 참조(`keyof`, 인덱스 시그니처, 문자열 키, `export *`), 미래 사용 의도, 외부 소비자 가능성을 모두 고려한다.
- 판단이 안 서면 P5 보류로 분류한다.

## 제약사항

- **보고 후 대기** — 6단계까지는 코드 수정 없음. 7단계는 사용자 지시 시에만.
- **P5는 수정 안 함** — 보고만 하고 사용자 판단에 맡긴다.
- **순차 점검** — 서브에이전트 병렬 호출하지 않음 (토큰 절약).
- **기존 Docs/Audit/ 파일 덮어쓰지 않음** — 항상 새 파일 생성.
- **워크스페이스 경계 존중** — 수정 시 core/frontend/backend 경계를 넘는 변경은 모든 소비자를 함께 수정.
