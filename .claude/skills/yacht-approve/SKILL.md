# 계획 승인 & 구현 검수

두 가지 모드를 지원한다:

- **plan** (기본): 구현 전 계획을 검토하여 승인
- **review**: 구현 완료 후 코드를 검수하여 승인

## 사용법

```
/approve              ← plan 모드 (직전 계획 전체 검토)
/approve plan 스코어보드  ← plan 모드, 특정 범위
/approve review       ← review 모드 (세션에서 구현한 전체 변경 검수)
/approve review 물리엔진 ← review 모드, 특정 범위
```

첫 번째 인자가 `review`이면 review 모드, 그 외는 plan 모드.

## Plan 모드 — 구현 전 계획 승인

### 전제 조건

- **구현 계획이 이미 수립**되어 있어야 한다 (Plan 모드, 태스크, 또는 대화 내 계획).
- 계획이 없으면 사용자에게 먼저 계획을 세우라고 안내하고 종료.

### 1단계: 검토 컨텍스트 수집

계획의 **영향 범위**에 해당하는 기존 코드를 파악한다:

- 계획이 수정/확장하려는 기존 파일 목록
- 해당 파일들의 현재 패턴 (구조, 네이밍, 인터페이스 등)
- 관련 모듈의 기존 구현 방식

### 1.5단계: 관련 문서 사전 검토

첫 회차 서브에이전트에게 `Docs/` 디렉터리를 읽고, 계획의 영향 범위와 관련된 문서(implementation_plan.md, task.md 등)가 있는지 판별하도록 지시한다.

- **관련 문서 있음**: 해당 내용을 검토 컨텍스트에 포함. 계획이 기존 아키텍처 결정과 충돌하지 않는지를 검토 기준에 추가.
- **관련 문서 없음**: 이후 회차에서 다시 참조하지 않는다.

### 2단계: 반복 검토 루프 (최대 8회)

**매 회차마다 반드시 새로운 opus 서브에이전트를 생성**한다.
기존 에이전트에 SendMessage로 재검토를 요청하는 것은 금지.

서브에이전트에게 전달할 프롬프트:

```
당신은 구현 계획 검토자입니다. 아래 계획을 검토하고 APPROVED 또는 REJECTED를 판정하세요.

## 검토 기준

### A. 계획 자체의 품질
1. 논리적 모순이 없는가
2. 누락된 단계가 없는가
3. 부작용(side-effect)을 간과하지 않았는가
4. 인터페이스/타입 변경이 모든 소비자에게 전파되는가

### B. 기존 코드베이스 패턴 준수
계획대로 구현했을 때, 기존 패턴을 무시하고 새롭게 만드는 부분이 없는지 확인:

1. **모노레포 구조**: frontend/backend/core 3개 워크스페이스 경계를 올바르게 존중하는가
2. **core 패키지 역할**: 게임 로직(점수 계산 등)은 core에, UI는 frontend에, 서버 로직은 backend에 위치하는가
3. **Zustand 스토어 패턴**: 상태 관리가 기존 gameStore.ts의 패턴을 따르는가
4. **React 컴포넌트 구조**: screens/ui/3d 디렉터리 구분을 올바르게 따르는가
5. **Three.js/R3F 패턴**: 3D 컴포넌트가 @react-three/fiber의 선언적 패턴을 따르는가
6. **물리 엔진 분리**: Rapier 3D 물리 연산이 서버 권위(server-authoritative) 원칙을 따르는가
7. **Socket.io 이벤트 구조**: 클라이언트-서버 통신이 기존 이벤트 패턴과 일관되는가
8. **결정론적 시뮬레이션**: 물리 동기화가 implementation_plan.md의 전략을 따르는가
9. **타입 공유**: frontend/backend 간 공유 타입이 core를 통해 전달되는가
10. **매직넘버 지양**: 공유 상수와 설정값을 사용하는가

### C. 기존 유사 구현과의 일관성
계획의 영향 범위에 이미 유사한 구현이 존재하면, 그 구현 방식과 일관되게 계획되었는지 확인.
"더 나은 방식"이라도 기존과 불일치하면 지적할 것. 일관성 > 개별 최적.

### D. 아키텍처 문서와의 충돌 (첫 회차만)
Docs/ 에서 계획 영향 범위와 관련된 문서를 확인하고:
1. 계획이 기존 아키텍처 결정을 위반하지 않는가
2. 동기화 전략(결정론적 물리 시뮬레이션)과 충돌하지 않는가
3. 서버 권위 원칙을 훼손하지 않는가

## 판정 형식

반드시 아래 형식으로 응답:

### 판정: APPROVED 또는 REJECTED

REJECTED인 경우:
- 각 문제를 번호 매겨 구체적으로 기술
- 문제 심각도: CRITICAL (구현 불가/버그 확실) / WARNING (패턴 위반/불일치)
- 개선 제안 포함

APPROVED인 경우:
- 주의사항이 있으면 NOTE로 첨부 (구현 시 유의할 점)

## 계획 내용
{계획 전체 텍스트}

## 기존 코드 컨텍스트
{영향 범위의 기존 파일/패턴 정보}
```

### 3단계: 판정 처리

- **APPROVED**: 루프 종료. 사용자에게 승인 결과와 NOTE(있으면) 보고.
- **REJECTED**: 지적사항을 반영하여 계획을 수정한 뒤, 2단계로 돌아가 새 서브에이전트로 재검토.
- **8회 소진**: 루프 종료. 미해결 지적사항을 사용자에게 보고하고 판단을 요청.

### 4단계: 결과 보고

```
## Plan 승인 결과
- 판정: APPROVED / REJECTED (N회차)
- 수정 이력: (있으면) 어떤 지적을 받아 무엇을 수정했는지
- 주의사항: (있으면) 구현 시 유의할 NOTE
```

## Review 모드 — 구현 후 검수

### 전제 조건

- 세션에서 **코드 구현이 완료**되어 있어야 한다.
- 구현된 변경이 없으면 사용자에게 안내하고 종료.

### 1단계: 변경 범위 파악

세션에서 구현한 변경 사항을 수집한다:

- 인자가 있으면 해당 범위만, 없으면 세션 내 전체 변경을 대상으로
- 변경/생성/삭제된 파일 목록
- 각 파일의 변경 내용 (diff 또는 전체 읽기)
- 관련 모듈의 기존 코드와의 관계

### 1.5단계: 아키텍처 문서 참조

- `Docs/implementation_plan.md`의 아키텍처 결정 중 변경 범위와 관련된 항목을 추출
- `Docs/task.md` 중 관련 있는 것을 식별
- 이 정보를 검수 컨텍스트에 포함

### 2단계: 반복 검수 루프 (최대 8회)

**매 회차마다 반드시 새로운 opus 서브에이전트를 생성**한다.

서브에이전트에게 전달할 프롬프트:

```
당신은 구현 코드 검수자입니다. 아래 구현을 검수하고 APPROVED 또는 REJECTED를 판정하세요.

## 검수 기준

### A. 코드 정확성
1. 인터페이스 불일치, 타입 오류가 없는가
2. 명백한 로직 오류가 없는가
3. 엣지 케이스 누락이 없는가
4. null/undefined 접근 위험이 없는가

### B. 아키텍처 문서 준수
1. implementation_plan.md의 아키텍처 결정을 위반하는 코드가 없는가
2. 서버 권위 원칙이 지켜지는가 (클라이언트에서 게임 결과를 결정하지 않는가)
3. 결정론적 물리 시뮬레이션 전략과 일치하는가

### C. 기존 코드베이스 패턴 준수
1. **모노레포 구조**: frontend/backend/core 워크스페이스 경계 존중
2. **core 패키지 역할**: 게임 로직은 core, UI는 frontend, 서버는 backend
3. **Zustand 스토어 패턴**: gameStore.ts의 기존 패턴 준수
4. **React 컴포넌트 구조**: screens/ui/3d 디렉터리 구분
5. **Three.js/R3F 패턴**: 선언적 3D 컴포넌트 패턴
6. **물리 엔진 분리**: 서버 권위 물리 연산
7. **Socket.io 이벤트 구조**: 기존 이벤트 패턴과 일관성
8. **결정론적 시뮬레이션**: 물리 동기화 전략 준수
9. **타입 공유**: core를 통한 타입 전달
10. **매직넘버 지양**: 공유 상수 사용

### D. 기존 유사 구현과의 일관성
변경 범위에 이미 유사한 구현이 존재하면, 일관되게 구현했는지 확인.

### E. 실사용 패턴에서의 부자연스러움
1. 실제 사용 흐름에서 어색하거나 예상과 다르게 동작할 코드가 없는가
2. 예약어·내장 식별자와의 이름 충돌이 없는가
3. 불필요한 복잡성(과도한 추상화, 사용되지 않는 파라미터 등)이 없는가

## 판정 형식

반드시 아래 형식으로 응답:

### 판정: APPROVED 또는 REJECTED

REJECTED인 경우:
- 각 문제를 번호 매겨 구체적으로 기술
- 문제 심각도: CRITICAL (런타임 오류/데이터 손실) / WARNING (패턴 위반/불일치) / NITPICK (사소한 개선)
- 해당 파일 경로와 라인 번호
- 구체적 수정 제안

APPROVED인 경우:
- 주의사항이 있으면 NOTE로 첨부

## 구현 내용
{변경된 파일 목록 + 각 파일의 변경 내용}

## 기존 코드 컨텍스트
{관련 모듈의 기존 패턴/구조}

## 관련 아키텍처 문서
{해당 범위와 관련된 아키텍처 결정 항목}
```

### 3단계: 판정 처리

- **APPROVED**: 루프 종료.
- **REJECTED**:
  - **CRITICAL/WARNING**: 본 에이전트가 직접 코드를 수정한다. 수정 후 새 서브에이전트로 재검수.
  - **NITPICK**: 보고만 하고 수정하지 않는다 (사용자 판단에 맡김).
- **8회 소진**: 루프 종료. 미해결 사항을 사용자에게 보고.

### 4단계: tsc 게이트

수정이 발생한 경우 각 워크스페이스에서 `tsc --noEmit`으로 타입 검사를 통과하는지 확인.
실패 시 수정하고 재검수 루프로 복귀.

### 5단계: 결과 보고

```
## Review 검수 결과
- 판정: APPROVED / REJECTED (N회차)
- 수정 내역: (있으면) 어떤 지적을 받아 무엇을 수정했는지
- NITPICK: (있으면) 수정하지 않은 사소한 개선 제안
- 주의사항: (있으면) NOTE
```

## 공통 제약사항

- 서브에이전트는 매회 **새로 생성**한다 (SendMessage 재사용 금지).
- **최대 8회** 루프 제한.
- Plan 모드는 코드를 수정하지 않는다.
- Review 모드는 CRITICAL/WARNING 지적에 대해서만 코드를 수정한다.
- README.md 등 문서 파일은 수정하지 않는다.
